delonix container

Ciclo de vida de containers: run, ps, start, stop, rm, exec, logs, inspect, stats, apply.

O grupo container é o dia a dia do runtime — o homólogo do docker container. Cada invocação é um processo efémero (sem daemon): o run faz clone() directo com os namespaces pedidos e o estado fica em JSON no $DELONIX_ROOT. Em rootless, o rootfs do container é uma cópia flat persistente — as escritas sobrevivem a stop/start, como no Docker.

Containers: run/ps/stop/rm/exec/logs

Usage: delonix container <COMMAND>

Commands:
  init     Inicializa um projecto com Delonixfile + manifesto — ficheiros JÁ PREENCHIDOS (imagens incluídas), prontos a usar sem editar nada
  run      Corre um container a partir de uma imagem (puxa se faltar)
  ps       Lista containers [alias: ls]
  start    (Re)arranca containers parados/crashados, reutilizando o rootfs persistente (as escritas feitas dentro do container sobrevivem, como no docker) e a mesma rede/portas/volumes do `run` original. Sempre detached
  stop     Pára um ou mais containers (SIGTERM, depois SIGKILL)
  rm       Remove um ou mais containers
  exec     Executa um comando dentro de um container a correr
  inspect  Mostra a spec completa de um ou mais containers (JSON do Store)
  stats    Uso de recursos (CPU/memória/PIDs) dos containers a correr — uma amostra e sai (sem stream). Sem IDs, mostra todos os que correm
  logs     Mostra os logs (containers detached)
  apply    Aplica os documentos `kind: Container` de um manifesto (idempotente por nome — um container já existente com esse nome não é recriado nem verificado quanto a drift de spec, ver `cmd::manifest`)
  help     Print this message or the help of the given subcommand(s)

Options:
  -h, --help  Print help

container run

Corre um container a partir de uma imagem (puxa se faltar)

Usage: delonix container run [OPTIONS] <IMAGE> [COMMAND]...

Arguments:
  <IMAGE>       Imagem (ex.: `alpine:3.19`)
  [COMMAND]...  Comando + argumentos (default: o ENTRYPOINT/CMD da imagem)

Options:
  -d, --detach                   Corre em segundo plano e imprime o ID
      --name <NAME>              Nome do container (default: `dlx-<id>`)
      --net <NET>                Rede: `host` (partilha a do host, default), `none` (netns isolado sem ligação), ou o NOME de uma rede criada com `delonix network create` [default: host]
  -v, --volume <VOLUMES>         Volume/bind mount, `nome:/destino[:ro]` ou `/host:/destino[:ro]`. Repetível
  -p, --publish <PUBLISH>        Publica uma porta, `hostPort:contPort[/tcp|udp]` ou só `porta`. Repetível. Com `--net host` (o default) muda o container para um netns próprio com NAT em userspace (slirp4netns, como o podman rootless); com `--net <rede>` publica pelo ingress (DNAT nft + hostfwd no slirp único)
      --privileged               Container privilegiado (todas as caps, seccomp off) — cargas de confiança
      --entrypoint <ENTRYPOINT>  Sobrepõe o ENTRYPOINT da imagem (o COMMAND passa a ser os argumentos deste binário; `--entrypoint ""` limpa-o e corre só o COMMAND)
      --rm                       Remove o container quando o processo terminar (em `-d`, um watcher destacado trata da remoção quando o container morrer)
      --restart <RESTART>        Política de reinício (só com `-d`): `no` (default), `on-failure[:max]`, `always`, `unless-stopped`. Um supervisor destacado (um por container, efémero — não há daemon) fica pai do container, captura o exit code real e reinicia-o conforme a política [default: no]
      --device <DEVICES>         Liga um device do host, `/dev/x[:/dev/y]`. Repetível. O `/dev` do container é um tmpfs com uma lista curada (null/zero/tty/...); isto acrescenta-lhe nós reais do host, como o `docker --device`
  -e, --env <ENV>                Variáveis de ambiente adicionais (`KEY=VAL`), repetível
      --label <LABELS>           Label (`KEY=VAL`), repetível — ex.: `io.x-k8s.kind.role=control-plane` activa a delegação de cgroup2 dedicada a nodes Kind (ver `setup_node_cgroup_ns`)
  -h, --help                     Print help

-p e a rede: com --net host (o default) o container muda para um netns próprio com NAT em userspace (slirp4netns — o modelo do podman rootless); com --net <rede> a porta é publicada pelo ingress (hostfwd no slirp único + DNAT nft), o caminho que permite trocar portas a quente sem parar o container. --net none recusa -p.

Exemplos

Servir nginx na porta 8080 do host (NAT userspace, sem root)
delonix container run -d --name web -p 8080:80 nginx
Correr numa rede criada pelo utilizador, publicando pelo ingress
delonix network create minha-rede
delonix container run -d --net minha-rede -p 8443:443 caddy
Shell descartável (remove-se sozinho à saída)
delonix container run --rm -e TERM=xterm alpine sh -c 'echo olá'
Sobrepor o ENTRYPOINT para depurar uma imagem
delonix container run --rm --entrypoint /bin/sh nginx -c 'nginx -t'

container ps

Lista containers

Usage: delonix container ps [OPTIONS]

Options:
  -a, --all    Inclui os parados/falhados
  -q, --quiet  Só imprime os IDs (para compor com `stop`/`rm`)
  -h, --help   Print help

Exemplos

Listar (alias `ls` também funciona)
delonix container ls -a
Compor com stop/rm
delonix container rm -f $(delonix container ps -aq)

container start

(Re)arranca containers parados/crashados, reutilizando o rootfs persistente (as escritas feitas dentro do container sobrevivem, como no docker) e a mesma rede/portas/volumes do `run` original. Sempre detached

Usage: delonix container start <IDS>...

Arguments:
  <IDS>...  

Options:
  -h, --help  Print help

Reusa a spec guardada (comando, env, volumes, rede, portas) e o rootfs persistente — ao contrário de rm+run, nada do que o container escreveu se perde.

Exemplos

Rearrancar um container parado, preservando o que foi escrito lá dentro
delonix container start web

container stop

Pára um ou mais containers (SIGTERM, depois SIGKILL)

Usage: delonix container stop [OPTIONS] <IDS>...

Arguments:
  <IDS>...  

Options:
  -t, --time <TIME>  Segundos até ao SIGKILL [default: 10]
  -h, --help         Print help

Exemplos

SIGTERM, e SIGKILL ao fim de 5s
delonix container stop -t 5 web db

container rm

Usage: delonix container rm [OPTIONS] <IDS>...

Arguments:
  <IDS>...  

Options:
  -f, --force  Força (mata se estiver a correr)
  -h, --help   Print help

Exemplos

Forçar remoção de vários

container stats

Uso de recursos (CPU/memória/PIDs) dos containers a correr — uma amostra e sai (sem stream). Sem IDs, mostra todos os que correm

Usage: delonix container stats [IDS]...

Arguments:
  [IDS]...  

Options:
  -h, --help  Print help

CPU%/memória/PIDs lidos do cgroup v2 do próprio container (resolvido por /proc/<pid>/cgroup, qualquer que seja a base delegada). Sem delegação de cgroup (rootless sem Delegate=yes), a memória cai para o VmRSS do init do container, marcada com ~.

Exemplos

Uma amostra de todos os que correm
delonix container stats

container apply

Aplica os documentos `kind: Container` de um manifesto (idempotente por nome — um container já existente com esse nome não é recriado nem verificado quanto a drift de spec, ver `cmd::manifest`)

Usage: delonix container apply [OPTIONS]

Options:
  -f, --file <FILE>  
  -h, --help         Print help

Exemplos

Aplicar só os `kind: Container` de um manifesto
delonix container apply -f delonix-manifest.yaml
